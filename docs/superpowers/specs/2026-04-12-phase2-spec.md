1. **Department-level access controls** (spec section 3: "deferred to Phase 2")

- Data source connectors beyond file system (Phase 3)

- Public-facing submission portal (not in spec)

## 3. Pillar 1: Department-Level Access Controls

### 3.1 Problem Statement

The master design spec (section 3) states: "Flat city-wide knowledge base (Phase 1). Department-level access controls deferred to Phase 2."

Currently, all authenticated staff see all requests, documents, and search results regardless of department. Phase 2 adds department scoping so staff only see data within their assigned department, while admins retain org-wide visibility.

### 3.2 Data Model Changes

No new tables required. Existing nullable `department_id` columns become enforced:

- `users.department_id` — Required for Staff, Reviewer, Read-Only roles. Nullable for Admin only.
- `records_requests.department_id` — Required on creation. Set from creating user's department.
- `documents.department_id` — Optional. When set, scopes document visibility. When null, visible to all departments (shared documents like city-wide policies).

### 3.3 Access Control Rules

| Role | Scope | Behavior |
|------|-------|----------|
| **Admin** | Org-wide | Sees all departments. Can create/edit departments. Can assign users to departments. No department_id required on user record. |
| **Staff** | Own department | Sees only requests and department-scoped documents in their department. Can search all shared (null department) documents. |
| **Reviewer** | Own department | Same as Staff, plus approve/reject within department. |
| **Read-Only** | Own department | View-only within department. |

### 3.4 Implementation

#### 3.4.1 Department CRUD API

```
POST   /departments/              — Create department (Admin only)
GET    /departments/              — List departments (Staff+)
GET    /departments/{id}          — Get department detail (Staff+, own dept or Admin)
PATCH  /departments/{id}          — Update department (Admin only)
DELETE /departments/{id}          — Soft-delete department (Admin only, must be empty)
```

#### 3.4.2 Scoping Middleware

Add `require_department_access()` dependency that:
1. Extracts `department_id` from the target resource (request, document).
2. Compares against `current_user.department_id`.
3. Allows if: user is Admin, OR user's department matches, OR resource has no department (shared).
4. Returns 403 if department mismatch.

This dependency composes with existing `require_role()`:
```python
async def require_department_access(
    resource_department_id: uuid.UUID | None,
    user: User = Depends(require_role(UserRole.STAFF)),
) -> User:
    if user.role == UserRole.ADMIN:
        return user
    if resource_department_id is None:
        return user  # shared resource
    if user.department_id != resource_department_id:
        raise HTTPException(status_code=403, detail="Access denied: wrong department")
    return user
```

#### 3.4.3 Query Scoping

All list endpoints that return requests or documents must add department filtering:

- `GET /requests/` — filter by `user.department_id` for non-admins
- `GET /documents/` — filter by `user.department_id` OR `document.department_id IS NULL`
- `GET /search/` — search results filtered to user's department + shared documents
- `GET /exemptions/flags/` — scoped by department through associated request

#### 3.4.4 Onboarding Extension

Extend the onboarding wizard with a department setup step:
1. Admin enters department names and codes (e.g., "Police" / "PD", "Finance" / "FIN")
2. Admin assigns initial users to departments
3. System sets `onboarding_status` to `complete` only after at least one department is configured

#### 3.4.5 Frontend Changes

- Department selector in admin user management (assign users to departments)
- Department filter dropdown on Requests and Documents list pages
- Department badge on request cards and document list items
- Department setup step in onboarding wizard
- Admin dashboard: per-department metrics

### 3.5 Migration

- Alembic migration to add NOT NULL constraint on `users.department_id` for non-admin roles (with CHECK constraint)
- Seed a default "General" department for existing users during migration
- All existing requests/documents assigned to "General" department

## 4. Pillar 2: Per-State Exemption Rule Expansion

### 4.1 Problem Statement

The spec requires per-state exemption rule configuration (section 9, Sub-Project 5). Currently 5 states are seeded (CO, TX, CA, NY, FL). Phase 2 expands to all 50 states.

### 4.2 Approach

Each state's open records statute defines specific exemption categories. The rules engine already supports three rule types:
- **regex** — Pattern matching (SSN, phone, email, credit card, etc.)
- **keyword** — Statutory phrase matching (e.g., "law enforcement investigation")
- **llm_prompt** — LLM-based secondary review prompt

PII regex rules (SSN, phone, email, credit card, DOB) are universal across all states. State-specific rules are statutory keyword patterns.

### 4.3 Implementation

#### 4.3.1 State Rule Research

For each state, identify:
1. The governing open records statute name and citation
2. Enumerated exemption categories
3. Key statutory phrases that indicate each exemption
4. Any state-specific PII categories (e.g., state ID formats, driver's license patterns)

#### 4.3.2 Rule Seed Script

Extend `backend/scripts/seed_rules.py` with all 50 states. Group by region for maintainability:

```
seed_rules.py
  ├── Universal PII rules (all states)
  ├── Western states (CO, CA, WA, OR, NV, AZ, NM, UT, MT, WY, ID, HI, AK)
  ├── Southern states (TX, FL, GA, NC, SC, VA, TN, AL, MS, LA, AR, KY, WV, MD, DE, DC)
  ├── Midwestern states (IL, OH, MI, IN, WI, MN, IA, MO, KS, NE, SD, ND)
  └── Northeastern states (NY, PA, NJ, CT, MA, RI, NH, VT, ME)
```

#### 4.3.3 State Configuration UI

- Dropdown to select active state on city profile
- Admin panel page showing all rules for the configured state
- Enable/disable individual rules
- Custom rule creation (regex or keyword) with audit logging
- Rule version tracking (spec section 6.6: "documentation of exemption rule sources with version tracking")

#### 4.3.4 Exemption Auditability Dashboard (Spec Section 6.6)

The spec requires:
- Flag acceptance/rejection rates by category, department, and time period
- Export flag accuracy data for external review
- Configuration interface for adjusting rules without code changes, with all changes logged

This dashboard extends the existing exemption endpoints:

```
GET /exemptions/dashboard/accuracy    — acceptance/rejection rates by category
GET /exemptions/dashboard/by-dept     — flag rates by department
GET /exemptions/dashboard/export      — CSV/JSON export of flag data
```

### 4.4 Deliverables

- 50-state exemption rule seed data
- State selection in city profile / onboarding
- Exemption auditability dashboard (spec 6.6)
- Rule version tracking

## 5. Pillar 3: Compliance Template Documents

### 5.1 Problem Statement

The master design spec (sections 6.4 and 6.5) requires five compliance template documents to ship with the product. The `disclosure_templates` table and API endpoints exist but contain zero seed data. No template document files exist in the repository.

### 5.2 Required Templates

Per spec sections 6.4 and 6.5:

| # | Template | Spec Section | Purpose |
|---|----------|-------------|---------|
| 1 | **Public AI Use Disclosure** | 6.5 | Statement cities post publicly disclosing their use of AI in records processing |
| 2 | **Response Letter Disclosure Language** | 6.5 | Boilerplate paragraph inserted into response letters disclosing AI involvement |
| 3 | **CAIA Impact Assessment Template** | 6.5 | Pre-filled Colorado AI Act impact assessment demonstrating the system is not high-risk |
| 4 | **AI Governance Policy Template** | 6.5 | Policy template based on GovAI Coalition, Boston, San Jose, Bellevue, Garfield County CO frameworks |
| 5 | **Data Residency Attestation** | 6.4 | Template for city IT directors attesting that all data remains on local hardware |

### 5.3 Implementation

#### 5.3.1 Template Files

Create `backend/compliance_templates/` directory with markdown source files:

```
compliance_templates/
  ├── ai-use-disclosure.md
  ├── response-letter-disclosure.md
  ├── caia-impact-assessment.md
  ├── ai-governance-policy.md
  └── data-residency-attestation.md
```

Each template includes:
- Placeholder variables (e.g., `{{CITY_NAME}}`, `{{STATE}}`, `{{EFFECTIVE_DATE}}`)
- Instructional comments explaining what the city should customize
- Source citations (which policy framework or regulation each section references)

#### 5.3.2 Seed Script

New seed script `backend/scripts/seed_templates.py` that:
1. Reads markdown files from `compliance_templates/`
2. Inserts into `disclosure_templates` table with appropriate `template_type`
3. Idempotent — can be re-run without duplicating

#### 5.3.3 Template API

Extend existing endpoints:

```
GET  /compliance/templates/                    — List all templates (Staff+)
GET  /compliance/templates/{id}                — Get template with content (Staff+)
GET  /compliance/templates/{id}/render         — Render with city profile variables filled in (Staff+)
POST /compliance/templates/                    — Create custom template (Admin only)
PUT  /compliance/templates/{id}                — Update template content (Admin only)
```

The `/render` endpoint substitutes placeholder variables from the city profile (city name, state, contact info, etc.).

#### 5.3.4 Frontend

- Compliance Templates page in admin panel
- Template viewer with rendered preview
- Download as PDF or DOCX
- Edit capability for admin customization

### 5.4 Template Content Guidelines

- Templates are starting points, not legal advice
- Each template includes a disclaimer: "This template is provided as a starting point. Consult your city attorney before adoption."
- CAIA Impact Assessment is pre-filled with CivicRecords AI-specific information (no auto-redaction, no auto-denial, human-in-the-loop enforcement) per the regulatory analysis
- All templates cite their source frameworks

The spec requires the admin panel to display model name, version, parameter count, license, and model card URL. The `model_registry` table exists with Phase 2 columns added. CRUD endpoints are needed:

```
GET    /admin/models/registry       — List registered models (Admin only)

PATCH  /admin/models/registry/{id}  — Update model info (Admin only)
```

The existing `GET /admin/models` endpoint (which queries Ollama) continues to show live model status. The registry provides the compliance metadata layer on top.
